# 什么情况下应该退回单 Agent

多 Agent 是个时髦词，BMAD 又给你装了 12+ 角色，会造成一种错觉——"我的项目是不是该全部改成多 Agent"。

不是。多 Agent 在某些场景下是负债，不是资产。学会"什么时候退回单 Agent"跟"什么时候上多 Agent"同样重要。

退回单 Agent 的场景大致有三类。

第一类是**任务本身没有真正的子问题**。比如"把这篇文章翻译成英文"——这是一个整体任务，硬拆成"先分析结构 agent + 再翻译 agent + 再润色 agent"只会让上下文在几个 agent 之间切来切去，每切一次 token 成本增加、一致性风险增加。一个 agent 在一个 context 里把事做完反而更好。判断原则是：**如果你把任务拆开之后发现每个子任务都需要"完整的上下文"才能做好，那这个任务就不该拆**。

第二类是**团队小、项目生命周期短**。多 Agent 的架构复杂度是真实成本——契约 spec 要写、角色协作要 review、失败传播要设计。如果你的项目就三个月、团队就两个人，这些成本摊下来比单 Agent 多走 1.5 倍以上。长寿项目才划算。判断原则是：**如果项目预期活不过六个月或团队不会超过三人，优先单 Agent，等项目证明了长期价值再考虑拆**。

第三类是**业务逻辑变动特别频繁**。多 Agent 系统最大的 hidden cost 是"改一个业务规则要改多个 agent + 改契约 spec"。如果你的业务每两周就要调一次策略，多 Agent 会把你耗死在同步各方的路上。单 Agent 改起来快得多。判断原则是：**业务变动频率超过架构稳定化速度时，多 Agent 反而会拖后腿**。

那什么时候真的值得上多 Agent？三个条件至少满足两个：

第一，任务里有**真正独立**的子能力，比如"感知 · 推理 · 行动"这种天然分层的。不是你硬拆的，是问题本身就长这样。第二，团队超过五人且项目周期超过六个月——人多项目长，分工带来的并行收益才能抵消协作成本。第三，需要不同模型处理不同环节——比如某些环节用 Claude 某些用 GPT某些用本地小模型，多 Agent 天然支持混合编排。

回到 ai-knowledge-base 项目本身。v1 其实是一个临界状态——三个 agent 看起来是多 Agent，但实际上每个都很轻、变动也不频繁，上 BMAD 是为了学习方法，不是业务必需。你在自己公司做决策时要把这个想清楚——不要因为学了 BMAD 就给所有项目都按个多 Agent 架构，那是为了锤子找钉子。

这也是 Week 3 最后我想留给你的问题：**你上周跑的那个 BMAD Party Mode 会议，如果你们团队真的这么开一场，是会帮你省时间还是浪费时间**？诚实回答它，你就知道自己项目该不该上多 Agent 了。
