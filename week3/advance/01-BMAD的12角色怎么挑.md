# BMAD 的 12+ 角色不是越多越好 · 怎么挑

学员装完 BMAD 第一眼会被吓到："它给我塞了十二个角色，PM、Architect、Developer、UX、Scrum Master、DevOps、Tech Writer……我是不是每次开发都要把这十二个角色都跑一遍？"

不用。一上来全上是最糟的用法。

我的经验是，任何一个项目在任何一个阶段，真正需要同时在线的角色**不会超过三个**。超过三个你就进入"委员会瘫痪"——每个人都有意见，没人负责决策，最后出来的 spec 是把所有人的顾虑都敷衍一遍的散文。

挑角色有一个很朴素的方法：问自己"如果这个决定出了问题，我最怕得罪谁"。通常你会想到两到三个人——比如"PM 会问我为啥没想到业务收益"、"Architect 会问我为啥选这个数据库"、"QA 会问我为啥没写回归测试"。这两到三个人对应的 BMAD 角色，就是你这次 review 要召集的。

不同阶段的组合也不同。早期需求阶段我常用 **PM + UX**——前者问业务收益、后者问用户能不能看懂。架构设计阶段我用 **Architect + DevOps**——前者管合理性、后者管运维成本。进入编码阶段我用 **Developer + QA**——前者管代码质量、后者管可测性。部署前我用 **QA + DevOps + Tech Writer**——三个视角把上线清单走一遍。

其他那些角色呢？Scrum Master、Product Owner、Security、Business Analyst——不是不用，是**"被需要时才用"**。比如你发现 review 里卡住了，三个人吵不出共识，这时候召一个 Scrum Master 角色进来说"这个问题不需要现在解决，进 backlog"——它的价值就出来了。平时没这个卡壳的问题，召进来只是增加噪音。

所以我给学员的建议是：**默认只装 BMAD 但平时只用三个角色**（PM / Architect / QA 是黄金三角），其他角色等"我真的需要 X 视角"时再叫。BMAD 的设计本来就是按需调用，不是全员群聊。

如果你看到有同事每次 review 都开 Party Mode 全角色上线——那不是在用 BMAD，是在演话剧。
