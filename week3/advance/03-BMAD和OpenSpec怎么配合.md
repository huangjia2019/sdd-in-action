# BMAD 和 OpenSpec 怎么配合 · 不要当成替代关系

学员问我："我上周刚装了 OpenSpec，这周又让我装 BMAD，它俩不是一回事吗？"

不是。它们是两件事，而且是互补的两件事。

OpenSpec 管的是"一个变更的生命周期"——从提 proposal 到写 spec、到设计方案、到拆任务、到实现、到归档。它关心的是**时间维度**，一个 change 从诞生到落地的全过程，每个阶段产出什么 artifact。它本质上是一个**流程工具**。

BMAD 管的是"一个问题的多视角 review"——PM 看业务、Architect 看架构、QA 看测试、UX 看交互。它关心的是**视角维度**，同一个问题被不同角色问一遍，暴露单视角看不到的盲点。它本质上是一个**思考工具**。

两者的正确配合方式是：**BMAD 帮你吵出共识，OpenSpec 帮你把共识落成 artifact**。

具体流程是这样的：你有一个模糊的想法，比如"给分析 agent 加一个质量阈值"。先用 BMAD 的 Party Mode 召集 PM + Architect + QA 三角色吵一轮，吵出一份会议纪要 `sessions/<date>-party.md`，里面有三条共识和两条悬而未决的问题。然后你用 `/opsx:new add-quality-threshold` 开一个新 change，把会议纪要的共识部分整合进 `specs/` 和 `proposal.md`，悬而未决的部分进 `tasks.md` 里作为"需要决策的任务"。代码实现完之后 `/opsx:apply`，验证完 `/opsx:verify`，归档 `/opsx:archive`——整个 change 生命周期走完。

半年后你回头看这个 feature，能看到两份资料：`openspec/archive/` 里是"最终实现的规范和代码"，`.bmad-core/sessions/` 里是"当时为什么这么做的讨论过程"。前者告诉你"是什么"，后者告诉你"为什么"。缺一不可。

有同学觉得两个工具一起用"太重了"。重不重取决于项目规模。单人项目用 BMAD 确实过重，一个 grill-me + OpenSpec 够了——因为你一个人需要的视角不多，真需要时自己切换一下角色就行。但两人以上的项目，尤其是不同人有不同"默认视角"的时候（前端工程师不会自觉站在运维视角想问题），BMAD 的角色切换就变成强制机制，很有用。

所以判断标准不是"项目大不大"，是"团队里的视角同质化程度高不高"。同质化高的团队需要 BMAD 来拉视角，同质化低的团队每个人天然带多个视角，BMAD 反而可能重复。

最后一个细节：两个工具的文件是分目录的。`.bmad-core/` 放角色讨论，`openspec/` 放 change artifact。不要混在一起。半年后你要找东西的时候，这种分层会帮你很多。
