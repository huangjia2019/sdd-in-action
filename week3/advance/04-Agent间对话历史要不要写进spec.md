# Agent 间的"对话历史"要不要写进 spec

多 Agent 项目上线一段时间后有同学问我："我的 agent 之间有传递历史消息——collector 会把今天的抓取结果连同'昨天 analyzer 的反馈'一起给 analyzer 看。这个'昨天的反馈'是不是要写进 spec？"

这是个挺好的问题，答案是**写，但不是直接写内容**。

对话历史这件事在多 Agent 系统里很微妙。短期来看它让 agent 变聪明——analyzer 看到"昨天这类条目 confidence 偏低"就会对今天的类似条目更谨慎。长期来看它是一个慢性污染源——历史记忆是 token 成本、是注意力稀释、是"半年前一次异常反馈至今还在影响判断"。

所以 spec 里要写的不是"历史内容长什么样"，是**历史消息的治理规则**。具体就三件事。

第一件是 **保留边界**。历史记忆保留多久？按条数（最近 N 条）还是按时间（最近 N 天）？两者配合（不超过 N 天也不超过 N 条）？这个策略要显式写在 spec 里，不能让代码里一个 `maxlen=50` 的魔法数字代替它。

第二件是 **清洗规则**。哪些历史消息值得留下？全部？还是只留"分歧特别大的 case"？还是"人工复核过的那些"？不同的清洗规则决定了 agent 会沿着什么方向演化。如果你没定义清洗规则，agent 会记得所有东西，然后被噪音淹没。

第三件是 **失效机制**。什么情况下历史要被清空？比如"模型升级了"、"业务规则变了"、"出现了大规模误判"——这些都是应该清空历史重新开始的时机。spec 里要显式列出这些触发条件，否则半年后谁也不敢动那份历史，生怕一动系统崩了。

这三件事合起来就是一份 **memory governance spec**。它和 agent 的 behavior spec 是分开的——因为 behavior 讲"agent 当下如何决策"，memory governance 讲"agent 凭什么拥有过去"。

有一个特别容易被忽略的隐患：**多 Agent 各自的历史记忆可能互相污染**。比如 analyzer 的历史反馈被 collector 也读到了，collector 以为"这是用户偏好信号"，实际上那只是 analyzer 的自我调整。这种跨 agent 的记忆串扰在 spec 里不写就没人注意，出了问题排查非常痛苦。所以 memory governance spec 最好还要包含一条"哪些 agent 可以看彼此的历史 · 哪些不能"。

总的来说，对话历史是多 Agent 系统的第四种 state（除了代码、配置、数据之外的第四种），它值得被当作一等公民管起来。草率对待它的项目，上线三个月之后会开始遇到"莫名其妙的判断漂移"这种诡异问题。
