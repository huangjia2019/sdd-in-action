# 从 grill-me 到 Superpowers · 四周的工具演化路径

四周下来你装了四个层级的工具：Week 1 的 grill-me（六行 markdown），Week 2 的 OpenSpec（七命令），Week 3 的 BMAD（十二角色），Week 4 的 Superpowers（接管全流程）。这个顺序不是随便排的，是有一条隐性逻辑的——**每一层都在补上一层的不足**。

grill-me 解决的是"AI 不问就开写"。它的能力边界是单次交互，问完一轮就结束。够吗？对单人小项目够，对长期项目不够——你写出来的 spec 散落在仓库各处，没人管它的生命周期。

OpenSpec 接管这个生命周期。一份 spec 从 proposal 到 archive 完整托管，能查哪些 change 在进行、哪些已归档、每个 change 的 4 份 artifact 长什么样。够吗？对单人或同质化团队够，对多角色团队不够——OpenSpec 的 spec 是"流程产物"，不带视角差异，PM/Architect/QA 看到的是同一份。

BMAD 补这个视角维度。十二个角色每个有自己的关注点，Party Mode 让多角色对同一个问题交叉拷问。够吗？对设计阶段够，对完整开发流程不够——BMAD 管"想清楚"，不管"做出来"。

Superpowers 接管"做出来"。从 brainstorm 到 merge 七阶段全程托管，TDD 强制嵌在其中，每一步都有 skill 兜底。够吗？对认真做产品的场景够，对原型探索过头。但这是工具定位，不是工具缺陷。

这条演化路径在我看来是**SDD 工具发展的必然形态**。L1 单点有用 → L2 持久化 → L3 流程化 → L4 全程化，每往上一层都是对"前面那层不够用"的回应。没有任何一层能取代其他层——你不能用 Superpowers 取代 grill-me，因为 grill-me 的 30 秒响应时间和零学习成本是 Superpowers 给不了的。

这也是为什么我反复跟学员说：**工具的层级要匹配问题的层级**。别一上来就 L4，L4 解决的问题你可能根本没有。也别死守 L1，L1 解决不了的问题你硬撑只会慢。

那下个项目你应该从哪一层开始？我的经验是几乎所有项目都从 L1 开始（grill-me），三周内观察自己的痛点，再决定要不要往上爬。这条路径几乎不会错——L1 即使最后被替代，那两周建立的"先想后做"的习惯不会消失。

四周走完你的工具栈不是越加越多就好。理想状态是：你知道每个工具解决什么问题，你的项目当下在哪一层，你能在下一个项目里**从零选一条最适合的路径**——可能跳过 BMAD，可能在 L3 就停下。这种选择能力比"装齐四层"更有价值。
