# grill-me + OpenSpec + BMAD + Superpowers 一起装会打架吗

四周下来你装了四个 SDD 工具。一个常见的问题是："它们同时存在会不会冲突？比如 grill-me 想拷问我，Superpowers 想 brainstorm，OpenSpec 想新建 change，谁先谁后？"

实际用下来的结论是：**不会打架，但你要主动决定优先级**。

不打架的原因是它们的触发机制不重叠。grill-me 是 SKILL.md 自动触发，靠用户输入里的关键词命中（比如 "grill me on..."）。OpenSpec 的 `/opsx:*` 是显式 slash 命令，必须你主动调。BMAD 的 `/agent` 和 `/party` 也是显式命令。Superpowers 的 `/brainstorm` `/write-plan` 同样是显式。所以"自动触发"的只有 grill-me 一个，其他三个都要你显式叫，天然不冲突。

但有几个细节需要主动管理。

**目录冲突要避开**。OpenSpec 用 `openspec/`，BMAD 用 `.bmad-core/`，Superpowers 用 `.plans/` 和 `.claude/sessions/`。这四个目录各管各的，加进 `.gitignore` 时不要漏。如果你把所有"AI 生成的中间文件"塞进同一个目录，三个工具会互相覆盖。

**工作流优先级要心里有数**。我自己的默认顺序是这样：

```
   grill-me   →   BMAD     →   OpenSpec  →   Superpowers
   (拷问)        (多角色 review)  (落 change)    (执行)
```

grill-me 在最前，因为它最便宜。能用 grill-me 解决的问题不需要 BMAD。BMAD 在 grill-me 之后，当单一视角不够时调出多角色。OpenSpec 把 BMAD 的共识落成 change 生命周期。Superpowers 在最后，处理 execute / TDD / merge 这种"动手"环节。

这个顺序不是必须的，是经验。你可以反过来——比如直接 `/brainstorm` 让 Superpowers 接管，跳过 grill-me。但跳过的代价是 brainstorm 的"四问"比 grill-me 浅一些（Superpowers 服务于流程速度，grill-me 服务于深度追问）。哪种选择都行，关键是你**有意识地选**，不是工具乱触发。

**别让一个会话里同时跑两个 L4**。BMAD 和 Superpowers 都是 L4 完整方法论。同时跑会出现"BMAD 在召集角色，Superpowers 也在催你 brainstorm"的尴尬。我的做法是：决策密集阶段（早期需求）开 BMAD，执行密集阶段（编码 / TDD / 部署）开 Superpowers。中间无缝切换，不重叠。

最后一个建议：**装的工具数量和你的项目复杂度匹配**。新项目第一周只装 grill-me。两周后引入 OpenSpec。一个月后视需求加 BMAD 或 Superpowers。从来不要"四个一起装"——那是学习 repo 的状态，不是生产 repo 的状态。生产 repo 应该精简到只有"当前真在用的"。

如果你下个项目把这四个全装上去结果发现一周只用 grill-me——那不是工具的错，是你装多了。装的时候问自己一句"这周我真的会用它吗"，能减少 90% 的工具污染。
