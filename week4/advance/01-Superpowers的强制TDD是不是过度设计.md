# Superpowers 的强制 TDD 是不是过度设计

学员装完 Superpowers 第一次跑就被 TDD 强制规则气到了——你说"先实现一下看看效果"，它说不行，要先写测试；你说"测试我等会儿补"，它说不行，源码动一行就拒绝；你说"这是探索性代码"，它说"探索性代码也要测试"。骂声一片。

我能理解这种反感。Superpowers 的 TDD 强制不是 nice-to-have，它是真的会拒绝你的 commit。第一次撞上去你会觉得这是工具在阻碍你，不是在帮你。

但用一两周之后多数人会改口。原因不是"我开始喜欢测试了"，是发现一件更深的事——**有 TDD 兜底之后，你敢改代码了**。这个差别非常大。

没 TDD 的时候改代码是赌博。你看着一个看起来烂的函数，想重构，但又怕改了某个隐藏调用方就炸——所以你不敢动。代码越积越烂，每个人都绕着走。这不是技术问题，是心理问题——人对未知风险天然规避。

有 TDD 的时候改代码是工程。你看到一个烂函数，先确认它有测试覆盖，然后大胆改。改完跑测试，绿了就行；红了就回滚或修测试。每一步都有反馈，没有"我不知道我刚才破坏了什么"的恐慌。

所以 Superpowers 强制 TDD 的真实价值不在覆盖率数字，在**心理安全感的预先投资**。它逼你前两周不爽，换来后面所有工作的从容。

那它是不是过度设计？我的判断分两种场景。

**生产代码场景：不过度**。任何要上线、要给用户用、要被同事接手的代码，TDD 强制是合理的。短期烦，长期省。Week 4 N14 让你回填 Week 1-3 的测试，就是在还这笔账。

**原型代码场景：过度**。你周末 hack 一个 demo，三天后就删——这种代码 TDD 是浪费时间。Superpowers 在原型场景下确实不友好。这时候你要做的不是骂工具，是切回 Claude Code 不带 Superpowers 跑这个 demo。原型完事再切回 Superpowers 做生产化改造。

工具有边界，不要用错场景。Superpowers 自己的定位是"keynote-level workflow"——它服务的是认真做产品的场景，不是周末玩具。理解了这个定位，TDD 强制就不再是过度，是恰到好处。

唯一让我有点保留的是 Superpowers 没提供"暂时退出 TDD 模式"的官方开关。这种小逃生口对工具的可用性很重要——再好的规则也总有少数场景需要例外。希望未来版本能加。
