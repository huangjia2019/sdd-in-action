# SDD 四周下来 · 你真正带走的是什么

四周到这里我们装了四个工具，写了十几份 spec，跑了大大小小几十个 mock screen。学员快毕业的时候经常会问："我这四周到底学到了啥？是这四个工具吗？"

不是。这四个工具半年后会过时——Superpowers 会改 API，OpenSpec 会再重构一次命令，BMAD 会合并某些角色，grill-me 可能被某个新 skill 取代。工具是流动的。

你真正带走的是**四件不会过时的东西**。

第一件是**"先想清楚再让 AI 做"的默认行为**。这件事听起来朴素，但对大多数人来说是要刻意建立的。Vibe Coding 的诱惑非常大——按一下回车 AI 就吐 200 行代码，谁不想啊。SDD 的训练价值不在工具，在让你**养成那个 30 秒的停顿**——动手前先打开 specs 文件夹看一眼，手痒时先写四个二级标题。这 30 秒的停顿能省你下游 30 个小时的返工。

第二件是**多视角拷问比单视角下结论靠谱**的认知。你这四周经历过 grill-me 一对一拷问，经历过 BMAD 三角色围攻，经历过 Superpowers 七阶段托管——它们的共同点是**强迫你跳出自己的视角**。一个人想问题永远会有盲区，AI 当对手能照见你的盲区。这个能力你下个项目还在用，下下个项目还在用，跟工具没关系。

第三件是**测试是行为契约，不是事后补丁**的工程素养。Week 4 的 Superpowers TDD 强制让你回填 Week 1-3 的测试，那两个小时你大概率骂过我。但你跑完会发现一件事——**有测试兜底之后，你敢改代码了**。这个安全感是不可逆的。一旦你尝过有测试时改代码的从容，你不会再回去裸改。

第四件是**可追溯性比当下效率更值钱**的判断力。spec 在 git 里、change 在 archive 里、BMAD 讨论在 sessions 里——这四周你产生了一堆"看起来多余"的文档。但三个月后某个深夜出 bug 的时候，你会感谢这些文档。当下多花 30 分钟换三个月后多睡两小时，这笔账永远划算。年轻工程师常觉得"代码就是一切"，资深工程师都知道**代码周围的元信息和代码本身一样重要**。

把这四件事写进你下一个项目的开发文化里，工具用什么都不要紧。Claude Code 也好 OpenCode 也好 Cursor 也好，来什么用什么；grill-me 也好 OpenSpec 也好 Superpowers 也好，按需要装。但这四件事是**底层**——它们决定了你未来五到十年的工程品味。

四周课程要结束的时候，最后留给你一个问题：你的**第一份 spec** 还在你的项目里吗？打开它，看一眼当时的笨拙，对比一下今天你能写出的版本。两个版本之间的差距，就是这四周真正给你的东西。

如果今天的版本比当时只是"更熟练"——那你只学到了表面。
如果今天的版本让你**重新思考了那个功能本身**——那你才真的学到了 SDD。

祝你下个项目顺利。
