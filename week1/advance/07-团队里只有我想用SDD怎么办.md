# 团队里只有我想用 SDD 怎么办

学员经常问我这个问题："我自己觉得 SDD 很好，但团队其他人还在直接 prompt AI，甚至觉得我写 spec 是浪费时间。我怎么让团队用起来？"

这个问题比任何技术问题都难十倍。但有实操路径，只不过这条路径反直觉。

我见过的推广失败的案例，九成都是因为用错了方式。错误的方式是"说服"——开会讲 SDD 有多好，发文章让同事读，写规范文件让大家遵守。这几件事我都试过，这几件事都没用。原因也简单：没人喜欢被别人说服，尤其是被一个"最近学了新东西的同事"说服。

真正有效的方式是"让同事自己羡慕"。一个人在团队里推方法论，推到最后只会变成那个"爱讲理论的人"；但一个人自己默默用得顺、交付快、bug 少，同事就会开始问你"你最近怎么 PR 质量这么高"。这是两种完全不同的局面。

这件事我分三个阶段去做。

第一阶段是"一个人先跑顺"，大概一两周。你自己的每一个 PR 都配一份 spec，不强求别人看。重点是让你的 PR 被 review 的速度变快——因为描述清楚、边界想过、验收标准明确，reviewer 少问十个问题，半小时搞定 PR。当同事发现这件事的时候，他们会开始好奇。

第二阶段是"被问起时再展开"，大概两到四周。同事会这样问你："你的 PR 描述怎么写这么清楚的？模板哪儿来的？"这是关键信号。这时候千万别开启一场 SDD 理论讲座，直接发他一份你的 spec 模板，说"你也试试，二十分钟写完一份，AI 生成代码更准"。一次只带一个人，不是全队同时。

第三阶段是"组内形成默认"，两三个月。当组里有两三个人开始用，PR 质量可见地提升，其他人会自发跟上。到这时候你再提议"新项目默认用 SDD 模板""code review 时检查有没有 spec""incident 报告用 spec 格式"——这些就变成团队共识，不是你在强推。

有几件事千万别做。别为了讲 SDD 专门开一场会；别在 tech sharing 里花一小时讲四层金字塔；别给老板写邮件申请"全员 SDD 培训"；别在 code review 里指责没写 spec 的同事。这四件事里任何一件都足以把你塑造成"那个爱讲流程的人"，之后你再怎么做都会被贴标签。

还有一个我觉得挺重要的技巧：**不要用"SDD"这个词**。"SDD"听起来像方法论派系，会触发防御反应。"规范驱动开发"听起来像流程主义。但如果你说"我最近发现让 AI 先按清单干活准多了"，同事会问你"清单怎么写"。你用术语等于关门，你用效果等于开门。

推广的速度也要有现实感。两三人小组大概一个月能有两三个人在用，三个月能变成默认模式。十人以上团队要慢得多，半年能覆盖一半就不错了，一年能形成组织共识已经算快的。但只要方向是往上走，慢没关系。

推广不是游说，是示范。自己先用顺，同事自己会来问。
