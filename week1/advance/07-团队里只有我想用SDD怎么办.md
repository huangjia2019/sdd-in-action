# 团队里只有我想用 SDD 怎么办

## 学员问题

> "我自己觉得 SDD 很好 · 但团队其他人还在直接 prompt AI · 甚至觉得我写 spec 是浪费时间。我怎么让团队用起来？"

这比技术问题难 10 倍 · 但有实操路径。

---

## 核心原则 · 不说服 · 只展示

**90% 的"推广失败"源于你用"说服"的方式**：

- 开会讲 SDD 有多好 ✗
- 发文章让同事读 ✗
- 写规范文件让大家遵守 ✗

**真正有效的是让同事"自己羡慕"**。

---

## 三阶段路径

### 阶段 1 · 一个人跑顺（1-2 周）

- 自己的 PR 每一个都配 spec
- 不强求别人看
- 重点：**让你的 PR 被 review 的速度变快**

当同事发现你的 PR 总是："描述清楚 / 边界想到了 / review 半小时搞定"—— 他们开始好奇。

### 阶段 2 · 被问起时再展开（2-4 周）

同事会这样问：

> "你的 PR 描述怎么写这么清楚的？模板哪儿来的？"

**这是你的信号**。

这时别讲 SDD 理论 · 直接发一份你的 spec 模板 · 说 "你也试试 · 20 分钟写完 · AI 生成代码更准"。

关键：**一次只带一个人**。不是全队同时。

### 阶段 3 · 组内形成默认（2-3 个月）

团队里有 2-3 人开始用 · PR 质量可见的提升 · 其他人开始自发跟。

到这时你再提议：
- 新项目默认用 SDD 模板
- Code review 时检查"有没有 spec"
- incident 报告用 spec 格式

**这些变成团队共识 · 不是你强推**。

---

## 不要做的事

**不要**：
- 开一整场会议专门讲 SDD
- 在 tech sharing 里用一小时讲金字塔
- 给老板写邮件申请"全员 SDD 培训"
- 在 code review 指责没写 spec 的同事

**要**：
- 先做给别人看 · 不先讲
- 被问起时再讲 · 讲完给一份模板
- 10 分钟讲完 · 让他自己回去试
- 一个月内复盘"你上次试了吗？感觉如何？"

---

## 一个真实数字

SDD 推广 2-4 人组团队：
- 1 周：1 个人（你）
- 1 个月：2-3 个人
- 3 个月：团队默认模式

推广 10+ 人的团队：
- 6 个月：半数人
- 1 年：组织级共识

**慢没关系 · 只要趋势向上**。

---

## 最后一个技巧 · 不要用 "SDD" 这个词

"SDD" 听起来像方法论派系 · 会触发防御反应。

你说 "规范驱动开发" · 同事会说 "又一个流程主义"。
你说 "先写 spec 再写代码" · 同事会说 "不就是瀑布吗"。
你说 **"我发现让 AI 先按清单干活准多了"** · 同事会问 "清单怎么写"。

**用术语等于关门 · 用效果等于开门**。

---

推广不是游说 · 是示范。自己先用顺 · 同事自己会来问。
