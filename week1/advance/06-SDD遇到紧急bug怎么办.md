# SDD 遇到紧急 bug 怎么办

学员问我一个很现实的问题：线上炸了，老板在群里 at 我，我还能按 Specify → Clarify → Implement 走流程吗？不走好像白学 SDD 了，走了感觉今晚下不了班。

这个问题不好答但答案出乎意料。先说结论：修复期在 30 分钟以内的，直接改代码，事后补 spec；修复期超过 30 分钟的，花十分钟先写最小 spec，反而更快。

为什么超过 30 分钟反而更快？这是一个反常识但很真实的现象。紧急 bug 真正拖时间的从来不是"修"这个动作本身，而是修错地方、引入新 bug、或者修了一半发现根因不对再回去重来。这三件事，每一件都能让一个本来半小时能完的事变成三小时。

那份十分钟的最小 spec 在帮你避免的就是这三件事。它只问三个问题：现象是什么（日志、用户报告、监控指标各是怎样）；受影响的范围多大（哪些用户、哪些功能、哪些版本）；修好之后用什么方式验证（具体步骤，不是"应该没问题了吧"）。就这三个问题。写完你大概率会发现：要么问题的根因比你第一感觉的深，要么你的第一方案会导致另一个功能出问题，要么你忽略了一个用户群。

十分钟写这份 spec，能避免一小时的返工和二次事故。净赚五十分钟。

实际操作我推荐 30-60-30 法则。一小时修复期这样分：前十分钟写现象、范围、验证三问；中间三十到四十分钟专心改代码加测试；最后十到二十分钟部署、监控、回填完整 spec。那份事后回填的完整 spec 才是这次 incident 留下的真正资产——三个月后类似 bug 再来一次，你手里有答案。

有一类例外，不需要 spec：纯配置改动。改个环境变量、翻转一个 feature flag、回滚一个版本——这些是运维级动作，不是代码级改动，SDD 管不着。但改完之后要写一份 incident report 放在 `knowledge/incidents/` 里，下次谁再遇到才有上下文。

最怕的反应是那句"这么急哪有时间写 spec，先改了再说"。这句话的潜台词是"流程会拖慢我"。但流程在紧急场景下恰恰是加速器。三次出错三次修，五小时后才下班——这是跳过流程的代价。十分钟 spec 然后一次成功一小时下班——这是流程的价值。

SDD 不是慢流程，是高效率流程。紧急场景下它反而更有价值。
