# 老项目逆向 · 改哪里，spec 就补到哪里

## 错误的第一反应

你面前一个 10 万行的老项目，没有 proposal.md，没有 design.md。
第一反应：

> "我要先给整个项目补写一套完整的规范文档"

这是错的。补完 2 周 · 代码一改规范又过时 · 结论 "SDD 不适合老项目" —— 错的不是 SDD，是策略。

---

## 正确策略 · 帕累托

**改哪里，规范就补到哪里。不改的地方，不补。**

```
这次改登录模块 → 只给登录模块补规范
下次改支付模块 → 只给支付模块补规范
bug 修到文件 X → 只给 X 涉及的函数补规范

半年后：
  频繁改动的核心模块 · 自然都有了规范
  很少改动的边缘模块 · 没规范也没事
  你从来没"专门补规范"—— 规范是日常开发的副产品
```

20% 代码承受 80% 改动 · 只给这 20% 补规范 · 用 20% 投入覆盖 80% 价值。

---

## OpenSpec 的 `/opsx:onboard`

OpenSpec 专为 Brownfield 设计的命令 · 一条命令给某个模块"增量"补 spec · 不重写不存在的部分。

```bash
# 给某个老模块补 spec
/opsx:onboard src/auth/

# 产出：
# openspec/changes/onboard-auth/
#   ├── proposal.md
#   ├── specs/auth-current-behavior.md
#   └── tasks.md
```

这样你得到的不是"强行补的过时文档"· 是"对当前代码行为的准确描述" · 下次改动就可以基于这份描述。

---

20% 投入覆盖 80% 价值。SDD 是开发的副产品 · 不是额外作业。
