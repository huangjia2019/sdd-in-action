# 老项目逆向 · 改哪里，spec 就补到哪里

这是学员问我最多的一个场景：我公司有一个跑了两三年的老项目，十万行代码，没有 spec，没有 design doc，README 还停留在两年前那个版本，原作者早走了。我现在想引入 SDD，从哪开始？

大多数人第一反应是——"我花两周时间给整个项目补一套完整的 spec 再说"。听起来合理。但这是新手错误。

我见过至少五个团队这么干过，无一例外地中途放弃。原因也一样：补了两周，代码改了三次，spec 已经对不上了；又补一周，发现补的那些 spec 里面一半是你臆想的"应该这样"，另一半是你从代码反推出来的"现在就是这样"——两边打架。最后得出一个假结论："SDD 不适合老项目。"

错的不是 SDD，错的是策略。

正确的做法朴素得让人失望：**改哪里，spec 就补到哪里。不改的地方，不补。**

这次你要改登录模块，那就花十分钟给登录模块写一份 spec。写的是它"现在的行为"加"你这次要改成什么"。改完代码，这份 spec 进 git。下次要改支付模块，给支付模块补一份。再下次修某个 bug，给 bug 涉及的那两个函数补一份。

半年之后回头看会发生一件挺神奇的事：那些频繁改动的核心模块，每一个都自然而然有了 spec，而且是新鲜的、跟代码同步的 spec；那些一年都没人碰的边缘模块，依然没有 spec，但它们也没人碰，所以没关系。

这不是偷懒。这是帕累托在 SDD 里的应用——项目里 20% 的代码承受 80% 的改动。只给那 20% 补 spec，用 20% 的力气覆盖 80% 的价值。剩下那 80% 的代码不值得你花时间补，因为你甚至不确定它们还有人在用。

具体怎么操作？Week 2 会讲 OpenSpec 的 `/opsx:onboard` 命令，它就是为这个场景设计的。你指定一个目录或模块，它帮你读代码、反推出"这段代码的 spec 应该长什么样"、写成一份 baseline 放在 `openspec/changes/` 下。这份 baseline 描述的是**当前代码的行为**，不是"理想中的代码应该是什么"——这个区别很重要。下次你要改这个模块，你改的是"当前行为"，你很清楚自己要把它从哪里改到哪里。

所以不要问"老项目怎么补 spec"，要问"下周我要改哪一块，我给那一块补 spec 就行"。SDD 是开发的副产品，不是额外作业。想通这一点，老项目和新项目没区别。
