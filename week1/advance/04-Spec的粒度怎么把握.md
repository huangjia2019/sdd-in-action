# Spec 的粒度怎么把握

## 学员问题

> "我一个功能的 spec 是写成 50 行还是 500 行？写少了 AI 产出不准，写多了自己都懒得维护。"

这是 SDD 落地的核心困惑。

---

## 三个层级的粒度

**粒度 1 · 愿景级** (vision) · 10-30 行
- 整个项目/产品在做什么
- 答"为什么存在" · 不答"怎么实现"
- 半年改一次

**粒度 2 · 功能级** (feature) · 50-150 行
- 一个用户可见的特性
- 包含边界 + 验收 + failure mode
- 每个 PR 对应一份 · 合入后移入 archive

**粒度 3 · 组件级** (component) · 100-300 行
- 一个模块/类/函数的契约
- 输入输出签名 + 不变量
- 只在"跨人协作 / 跨 Agent 协作"时才需要

---

## 实操建议

**多数人应该从粒度 2 开始**。

- 粒度 1 太高 · 写了落不到代码
- 粒度 3 太低 · 会变"函数文档" 失去 spec 的方向感
- 粒度 2 是"一个 PR 的契约" · 粘度刚好

---

## 一个经验法则

**spec 的每一行都应该是可以被"违反"的**。

- "本模块处理用户数据" —— 违反不了 · 删掉
- "本模块不存储明文密码" —— 可以被违反 · 保留

如果你的 spec 里一堆"描述性"句子 · 那是 README 不是 spec。
spec 的每一行都应该可验证或可验证其否定。

---

## spec 的三种 Smell

看到这些要警惕：

| Smell | 症状 | 对策 |
|-------|-----|------|
| **散文 spec** | 一段话 300 字没有 bullet | 强制按 4 H2 重写 |
| **实现细节 spec** | "用 Redis 做缓存" | 问"为什么" · 改成契约（"查询 p99 < 100ms"） |
| **万能 spec** | 一份 spec 300 行想管所有 | 拆成 3-5 份功能级 spec |

---

切分粒度没有黄金答案 · 但"一个 PR 一份 spec"是个好默认值。
