# Spec 的粒度怎么把握

写过两三份 spec 的学员都会遇到一个问题：我这份 spec 到底应该写多长？写五十行 AI 看了不够用，写五百行我自己都懒得维护。这个问题没有标准答案，但有一个大致的分层可以帮你定位。

我把 spec 分成三种粒度去看。最大一种叫愿景级，通常十到三十行，答的是"这个项目为什么存在"，不答"怎么实现"。它半年才改一次，是整个项目的北极星。中间一种叫功能级，五十到一百五十行，对应一个用户可见的特性——里面包含边界、验收标准、失败模式。理想情况下一个 PR 对应一份功能级 spec，PR 合入了 spec 就进 archive。最小一种叫组件级，一百到三百行，描述一个模块或类或函数的契约，里面有输入输出签名、不变量、调用约定。组件级 spec 只在跨人或跨 Agent 协作的场景下才值得写，一般项目用不到。

多数人应该从功能级开始。愿景级太高，写了落不到代码；组件级太低，会变成一份加长版的函数文档，失去 spec 的方向感。功能级刚好——一个 PR 的契约，一份 spec，粘度合适。

那份 spec 里每一行该写什么？我有一个经验法则：**每一行都应该是能被"违反"的**。"本模块处理用户数据"这种句子违反不了，它只是描述，删掉没事；"本模块不存储明文密码"就不一样，它可以被违反，你一眼就能看出哪个版本的代码违反了它。spec 的每一行都该是能验证或能验证其否定的，剩下的那些描述性句子属于 README，不属于 spec。

有三个常见的 smell 你要警惕。第一个是散文 spec：一整段话三百字没有 bullet，读起来像小作文。这种要强制按四个二级标题重写。第二个是实现细节混入 spec："本服务用 Redis 做缓存"——为什么用 Redis 不在 spec 里，spec 里该写的是契约（"查询 p99 延迟低于 100ms"），用什么实现那是实现的事，会变，变了 spec 不该跟着改。第三个是万能 spec：一份 spec 写了三百行想管所有事。这时候应该拆——按功能切三到五份功能级 spec，每份管一件事。

切分粒度这件事没有黄金答案，但"一个 PR 一份 spec，写五十到一百五十行"是个好默认值。当你开始觉得这个默认值在某个场景下不够用或太啰嗦，你其实已经有判断力了——那就按你的判断来。
