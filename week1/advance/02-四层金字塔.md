# SDD 工具生态四层金字塔 · 你该用哪层

学员群里最常问的问题是"SDD 工具那么多我该装哪个"。这个问题没有一个直接答案，因为它默认了一个错误前提：所有工具都在解决同一个问题。其实不是。

我把这些工具分成四层去看。最底层我叫"原子技能"，代表是 grill-me、tdd skill 这些——一个 SKILL.md 就六行，功能极其单一，比如"在 AI 写代码前逼它先追问"。往上一层是"上下文持久化"，代表是 AGENTS.md、Trellis 这些——解决多次会话 AI 忘了项目规矩的问题。再往上是"规范框架"，代表是 OpenSpec、Spec-Kit——管的是 spec 这份文档本身怎么生成、怎么版本化、怎么 verify。最顶层是"完整方法论"，代表是 Superpowers 和 BMAD——它们不是管某一件事，而是接管你从头脑风暴到代码合入的整个流程。

大多数学员犯的错不是选错了工具，而是在 L1 的需求上选了 L4 的工具。你想要的只是"AI 一上来别直接写代码，先问我几个问题"——这是一个很轻的 L1 需求，grill-me 那六行 Markdown 就解决了；结果你装了 BMAD，十二个角色十二个配置文件，学习成本一整天，跑完你会觉得"SDD 好累"。不是 SDD 累，是你在一个六行 Markdown 能解决的问题上压了一座山。

怎么判断自己是哪一层？看你当下最痛的是什么。如果痛的是"AI 不问我就开始瞎写"，你要的是 L1。如果痛的是"这次会话教给 AI 的项目规矩，下次它又忘了"，你要的是 L2。如果痛的是"需求和代码中间那段——设计方案、任务拆分、验收标准——每次都乱七八糟没法复用"，你要的是 L3。如果痛的是"我希望从想一个功能到合入 main 之间，整个流程是规范的、自动化的、可审计的"，那才是 L4。

行动营的工具路线也是按这个层级来的。Week 1 只上 L1，grill-me、prd-to-plan、write-a-skill 三个原子技能，从最简单的开始让你有肌肉记忆。Week 2 上 L3 的 OpenSpec，把 spec 这件事从"手写 markdown"升级到"有命令有版本有 verify"。Week 3 和 Week 4 才上 L4 的 BMAD 和 Superpowers——到那时候你已经写过四周 spec 了，再装一个完整方法论框架才知道它在帮你省什么。

顺序很重要。一上来就 L4 是灾难，因为你不知道框架在帮你省什么、什么时候可以自己来。反过来也一样——如果你已经做大团队需要治理了，还在手写 markdown，那也是在浪费你自己的时间。工具的层级必须匹配问题的层级，错配出来的 95% 的烦恼都是这里来的。
