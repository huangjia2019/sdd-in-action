# 练习：Vibe Coding 问题诊断

> 对应《规范驱动开发》第 1 章

## 目标

用这份检查清单审视你最近用 AI 完成的一个项目，诊断 Vibe Coding 的五个致命问题在多大程度上存在。

## Vibe Coding 诊断清单

对以下每个问题回答"是"或"否"：

### 需求蒸发

- [ ] 项目有正式的需求文档（不是对话记录）吗？
- [ ] 主要的技术决策（框架选型、架构方案）被记录在某个持久化的地方了吗？
- [ ] 如果有人问"为什么选了 X 而不是 Y"，你能在 30 秒内找到答案吗？

### 上下文漂移

- [ ] 项目中不同模块的代码风格是否一致？
- [ ] 错误处理方式在全项目中是否统一？
- [ ] 数据模型/接口定义在不同文件中是否一致？

### 不可审查

- [ ] 如果要做代码审查，审查者能看到每段代码的设计意图吗？
- [ ] PR 描述中是否包含设计决策的背景，而不仅是"实现了 XX 功能"？

### 不可复现

- [ ] 如果要从零重建这个项目，现有文档是否足以指导？
- [ ] 项目的架构决策是否可以被新人理解（不需要问原作者）？

### 不可维护

- [ ] 你有信心在三个月后修改这段代码而不引入新 bug 吗？
- [ ] 如果你离开这个项目，接手的人能独立维护吗？

## 评分

- **0-3 个"是"**：重度 Vibe Coding。建议从第 2 章开始认真学习 SDD。
- **4-7 个"是"**：中度 Vibe Coding。有一些好习惯，但缺乏系统性。
- **8-11 个"是"**：轻度 Vibe Coding。你已经有了工程化意识，SDD 会帮你系统化。
- **12 个"是"**：你可能已经在做 SDD 了，只是没有给它起名字。

## 下一步

完成诊断后，带着你的评分继续阅读第 2 章。第 2 章将介绍 SDD 的核心思想，你会看到每一个"否"的答案对应的解决方案。
