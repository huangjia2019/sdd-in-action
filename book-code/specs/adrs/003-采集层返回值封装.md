# ADR-003：采集层返回 CollectResult 而非裸列表

## 状态

已采纳

## 背景

在实现 Task 6（日报生成模块）时，发现一个接口缺陷：采集层接口返回 `list[Record]`，无法区分两种语义截然不同的场景：

1. **正常采集，但该时间段内确实没有数据** → 返回空列表
2. **API 调用失败，采集未成功** → 也返回空列表

日报生成模块的验收标准要求：数据源采集失败时，日报中标注"数据获取失败"。但在现有接口下，生成模块收到空列表后无法判断应该显示"今日无记录"还是"数据获取失败"。

这是一个**设计层面的缺陷**，不能在代码层打补丁解决。

## 选项

### 选项 A：返回裸列表 + 异常

采集层在失败时抛出异常，调用方通过 try/except 区分。

- 优点：接口签名不变
- 缺点：异常作为控制流违反了 design.md 的错误处理策略（优雅降级原则要求采集失败不中断流程）

### 选项 B：返回 CollectResult 封装对象

定义一个包含 success 标志、记录列表和错误信息的封装类型。

- 优点：语义明确，调用方可根据 success 标志决定行为
- 缺点：所有采集模块和调用方需要适配新类型

### 选项 C：返回 (list[Record], Optional[str]) 元组

返回二元组，第二个元素为错误信息（None 表示成功）。

- 优点：简单，不需要定义新类型
- 缺点：元组语义不如命名字段清晰

## 决策

采用**选项 B**：引入 `CollectResult` 封装类型。

```python
@dataclass
class CollectResult:
    success: bool
    records: list  # list[CommitRecord] / list[TaskRecord] / ...
    error_message: str | None = None
```

## 理由

1. **语义清晰优先**：CollectResult 的字段名自解释，比元组和异常更容易理解和维护
2. **符合优雅降级原则**：调用方检查 `result.success`，不需要 try/except
3. **改动范围可控**：只需修改 3 个采集模块的返回值和日报生成模块的调用方
4. **可扩展**：未来可以在 CollectResult 中添加更多元信息（如采集耗时、数据条数等）

## 影响

- 修改 `collector/github.py`、`lark_task.py`、`lark_msg.py` 的返回类型
- 修改 `generator/formatter.py` 的调用逻辑
- 更新 `specs/contracts/data-models.md` 新增 CollectResult 定义
- 更新 `specs/tasks.md` Task 3-6 的验收标准
