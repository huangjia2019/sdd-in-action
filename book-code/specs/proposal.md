# 智能日报生成器

## 1. 背景与目标

### 1.1 背景
团队每天需要花 15-30 分钟手写日报，内容包括当天的代码提交、任务进展和协作沟通。这个过程重复、低效，且经常因为遗忘而不完整。团队 Leader 需要了解每个人的工作进展，但手写日报的信息密度低、格式不统一，阅读体验差。

### 1.2 目标
构建一个自动化工具，从团队现有的协作平台中收集当天的工作数据，自动生成结构化的日报，并推送给相关人员。

### 1.3 目标用户
- **主要用户**：5-15 人的研发团队成员（日报的数据来源）
- **主要读者**：团队 Leader / 项目经理（日报的消费者）

## 2. 功能范围

### 2.1 做什么（本期）

#### 数据采集
- 从 GitHub/GitLab 获取当天的代码提交记录（commit message + 变更统计）
- 从飞书/Jira 获取当天的任务状态变更（新建、进行中、已完成）
- 从飞书群消息中提取与项目相关的讨论摘要

#### 日报生成
- 将采集到的数据按"代码提交 → 任务进展 → 协作沟通"三部分组织
- 为每个团队成员生成独立的日报段落
- 输出格式：Markdown（可转为 HTML 邮件）

#### 推送通知
- 每个工作日 18:00 自动触发日报生成
- 通过邮件发送 HTML 格式日报给 Leader
- 通过飞书机器人将 Markdown 日报发到团队群

### 2.2 不做什么（本期明确排除）
- **不做**智能摘要和 AI 总结（本期只做数据整理，不做语义理解）
- **不做**移动端 App
- **不做**历史日报的搜索和统计分析
- **不做**个人日报的编辑和补充功能
- **不做**多团队 / 多租户支持

### 2.3 技术约束
- 后端语言：Python 3.11+（团队技术栈）
- 部署环境：公司内网服务器（Docker 容器）
- 外部依赖：GitHub API、飞书开放平台 API、公司邮件服务（SMTP）
- 运行方式：定时任务（cron），不需要常驻服务
- 数据存储：本地 SQLite（存储日报历史，不需要远程数据库）

## 3. 验收标准

### 3.1 功能验收
- [ ] 能正确获取指定 GitHub 仓库过去 24 小时内所有成员的 commit 记录
- [ ] 能正确获取飞书项目中当天状态变更的任务列表
- [ ] 能从飞书群消息中提取包含指定关键词的讨论记录
- [ ] 生成的日报包含三个部分：代码提交、任务进展、协作沟通
- [ ] 日报 Markdown 格式正确，可直接渲染为可读的 HTML
- [ ] 邮件发送成功，Leader 收到 HTML 格式日报
- [ ] 飞书机器人成功将日报发送到指定群

### 3.2 性能验收
- [ ] 单次日报生成（5 人团队）耗时 < 60 秒
- [ ] 数据采集失败（某个 API 超时）不影响其他数据源的采集
- [ ] 采集失败的数据源在日报中标注"数据获取失败"而非静默跳过

### 3.3 边界场景
- [ ] 某人当天没有任何提交/任务/消息时，日报中显示"今日无记录"
- [ ] GitHub API 限流时，等待后重试（最多 3 次）
- [ ] 飞书 token 过期时，自动刷新 token
- [ ] 非工作日（周末、节假日）不触发日报生成

## 4. 术语定义

| 术语 | 定义 |
|------|------|
| 日报 | 系统自动生成的每日工作总结，包含代码、任务、沟通三部分 |
| 成员 | 在系统中配置了 GitHub 和飞书账号映射关系的团队成员 |
| 数据源 | 提供工作数据的外部平台（GitHub、飞书任务、飞书消息） |
| 采集窗口 | 每次采集覆盖的时间范围，默认为当天 00:00 至执行时刻 |
