# 成本超标的 spec 应该长什么样

很多同学第一次写成本相关的 spec 会这样写：

> ## 要做什么
> - 监控每次 LLM 调用的 token 花费
> - 每月总花费不超过 $100

然后就没有了。这份 spec 在月底账单爆掉之前不会告诉你任何事，爆掉之后也不会告诉你任何事。它是许愿，不是规格。

成本类 spec 和普通功能 spec 有一个本质区别：它是**横切关注点**，管的不是"一个功能"，而是"所有调用 LLM 的地方都要遵守"。所以它的结构不能照抄功能 spec 的四个二级标题，得额外加三段才完整。

第一段是 **预算**。预算必须分层，至少三层——每次调用、每天、每月。"每次 < $0.02 · 每天 < $5 · 每月 < $100" 这样写。分层的原因是当你发现月超标的时候，已经晚了；每日超标能提前 30 倍预警；每次超标能立刻抓住异常行为（比如某次 retry 风暴）。

第二段是 **告警**。告警要按"预算消耗比例"写，不是按"绝对数字"。"日预算消耗到 80% 时发 warning · 100% 时发 critical · 120% 时停 pipeline"——这样写的好处是无论预算怎么调，告警规则不用改。不要写"日花费超过 $4 告警"，这种硬编码数字半年后你自己都不记得出处。

第三段是 **降级策略**。超标怎么办？这是 99% 的成本 spec 都漏写的地方。我见过项目超预算的处理是"发 Slack 给 CTO"——等 CTO 醒了钱已经烧完了。降级策略应该写成可自动执行的：比如"消耗超过 80% 时自动切换到 Claude Haiku（便宜模型）"、"超过 100% 时跳过当日非关键 agent"、"超过 120% 时停止 pipeline 并发告警"。每一条都应该是代码可以直接翻译的指令，不是"人工判断"。

一份完整的成本 spec 长这样：

```markdown
## 预算（分三层）
- 单次调用：< $0.02（异常值报警）
- 每日：< $5（硬上限）
- 每月：< $100（季度复盘调整）

## 告警（按比例）
- 日消耗 50% · 记 info log · 不打扰
- 日消耗 80% · Slack warn 到 #devops
- 日消耗 100% · Slack critical + 邮件 team lead
- 日消耗 120% · PagerDuty + 停 pipeline

## 降级策略（自动执行）
- 80%-100% · 切便宜模型（Haiku 替 Sonnet）
- 100%-120% · 跳过非关键 agent（organizer 可跳）
- 120%+ · 停 pipeline · 写 incident/{date}.md
```

这份 spec 配合 `/opsx:apply` 生成的 cost-tracker 代码，就是一个能真正管住账单的系统。

成本 spec 最容易犯的错是"漏预算的粒度"和"漏降级策略"。粒度漏了你只能等月末看账单哭，降级策略漏了你的 tracker 只是一个记账本，不是管控手段。
